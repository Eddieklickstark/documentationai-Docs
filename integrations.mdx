## Clerk

Clerk ist für interne Admin-Authentifizierung und optionales Kundenlogin per Microsoft zuständig.

Produktion:

- Frontend API: `https://clerk.addsales.io`
- Redirect URI für Microsoft OAuth: `https://clerk.addsales.io/v1/oauth_callback`
- Convex nutzt `CLERK_JWT_ISSUER_DOMAIN=https://clerk.addsales.io`.
- Vercel braucht `VITE_CLERK_PUBLISHABLE_KEY` für die Vite-App.
- Microsoft OAuth nutzt die Application Client ID und den Secret Value aus Azure; die Secret ID wird in Clerk nicht verwendet.
- Das Frontend nutzt Clerk Redirect-Auth und rendert eigene Account-Steuerelemente. Clerk UI-Komponenten werden nicht gemountet.
- In Clerk muss ein JWT Template mit dem Namen `convex` existieren. Dieses Template muss den Audience-Wert `convex` setzen, weil `convex/auth.config.ts` `applicationID: "convex"` erwartet.

## Convex

Convex speichert:

- Kunden, Sprints und Ziele
- Dashboard-Invites, Magic Links und Sessions
- Leads, Sequence Memberships, Activities und Termine
- HubSpot Sync Runs und Snapshots

Der stündliche HubSpot-Sync läuft in `convex/crons.ts`.

## HubSpot

HubSpot ist die Quelle für:

- Contacts
- Sequences
- Calls
- Emails
- Meetings
- Owners und Users

Der Sync nutzt sowohl kundenbezogene AddSales-Properties als auch CRM Engagements.

## Resend

Resend versendet Dashboard-Invites und Magic Links. Provider-Events werden in `emailEvents` protokolliert.

Convex Production braucht dafür:

- `RESEND_API_KEY`
- `RESEND_FROM_EMAIL`, aktuell `AddSales <dashboard@dashboards.addsales.io>`

Wenn Resend den API Key mit `401 Unauthorized` ablehnt, werden Einladungen zwar in Convex angelegt, aber keine E-Mail verschickt. In diesem Fall muss ein neuer gültiger API Key aus dem richtigen Resend Account in Convex Production gesetzt werden. Nach dem Setzen sollte ein Dashboard-Invite oder Magic Link erneut getestet werden.

Aktueller Stand: Der neue Production API Key ist in Convex gesetzt und in 1Password im Vault `Founder` als `AddSales Dashboard Resend API` gespeichert. Resend akzeptiert den Key. Die Domain `dashboards.addsales.io` ist in Resend vorhanden, aber noch nicht verifiziert.

DNS-Records für Resend:

- TXT `resend._domainkey.dashboards.addsales.io` mit `p=MIGfMA0GCSqGSIb3DQEBAQUAA4GNADCBiQKBgQDlhJXaCH4yNag1xga/14Yf/4QnyrP8g1KbzlFFmCweUucA6CqZqA600S6/SGxsexUA87iej9oudwr8sbR7dIWxJbKMzfLtgNKgs+n5jpJlVMQImetJIfGCE83hNE26OPdr5UmrZ+sozosfhK/UG9GGz9zBbJvrZllYP8IeTRxJ2QIDAQAB`
- MX `send.dashboards.addsales.io` auf `feedback-smtp.eu-west-1.amazonses.com`
- TXT `send.dashboards.addsales.io` mit `v=spf1 include:amazonses.com ~all`

Nach dem Setzen der DNS-Records muss die Domain in Resend verifiziert werden. Erst danach können Mails von `dashboard@dashboards.addsales.io` zuverlässig versendet werden.

## Vercel

Das Frontend wird als Vite-App gebaut und über Vercel ausgeliefert. Die Convex-URL und der Clerk Publishable Key sind Frontend-Environment-Variablen.
